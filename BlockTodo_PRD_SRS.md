## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 서비스명 | BlockTodo |
| 목적 | Ethereum Sepolia 테스트넷에서 동작하는 블록체인 기반 To-Do List |
| 스마트 컨트랙트 | Solidity 0.8.x |
| 블록체인 네트워크 | Ethereum Sepolia Testnet |
| 데이터 저장 방식 | 완전 온체인 (All data stored on-chain) |
| 프론트엔드 | React + ethers.js |
| 지갑 | MetaMask |

### 1.1 핵심 가치 제안

- **불변성**: 체인에 기록된 할 일은 위변조 불가
- **자기 주권**: 서버 없이 지갑 주소 = 계정, 데이터 소유권이 사용자에게
- **투명성**: 모든 생성/수정/삭제 이력이 블록체인에 공개적으로 기록됨

## 2. PRD — 제품 요구사항 문서

### 2.1 목표 및 배경

| 구분 | 내용 |
|------|------|
| 배경 | 기존 To-Do 앱은 서버 의존적이며 데이터 소유권이 플랫폼에 귀속됨 |
| 목표 | 블록체인 기반으로 완전한 데이터 자기 주권을 실현하는 To-Do 서비스 구현 |
| 성공 지표 | Sepolia에서 CRUD 전 기능 정상 동작, MetaMask 연동 완료 |

### 2.2 타겟 사용자

| 페르소나 | 설명 |
|----------|------|
| 블록체인 개발 학습자 | Web3 연동 프론트엔드 개발 학습 목적 |
| 1인 개발자/사이드프로젝트 실험자 | 탈중앙화 앱 구조 실습 |
| MetaMask 지갑 보유 Ethereum 사용자 | 기존 지갑으로 바로 사용 |

### 2.3 핵심 기능 목록

#### F-01. 지갑 연결 (Wallet Connect)
- MetaMask를 통해 지갑 주소로 로그인
- 연결된 주소, 네트워크(Sepolia) 표시
- Sepolia 외 네트워크 연결 시 경고 및 자동 네트워크 전환 요청

#### F-02. To-Do 항목 관리 (CRUD)
- **추가**: 제목, 설명, 카테고리, 우선순위, 마감일을 입력하여 온체인 등록
- **조회**: 내 지갑 주소에 속한 모든 To-Do 항목 목록 표시
- **수정**: 제목, 설명, 카테고리, 우선순위, 마감일 변경
- **삭제**: 선택한 To-Do 항목을 온체인에서 삭제 (논리적 삭제 플래그 처리)
- **완료 토글**: 완료/미완료 상태 전환

#### F-03. 카테고리 관리
- 카테고리 생성, 조회, 삭제
- 카테고리는 지갑 주소에 귀속
- 기본 카테고리: `일반(General)` (컨트랙트 초기화 시 자동 생성)

#### F-04. 우선순위
- 3단계: `HIGH`, `MEDIUM`, `LOW`
- To-Do 생성/수정 시 선택 가능

#### F-05. 필터 및 정렬
- 필터: 완료 여부, 카테고리, 우선순위
- 정렬: 생성일 순, 마감일 순, 우선순위 순

#### F-06. 트랜잭션 피드백
- 트랜잭션 제출 후 Pending 상태 표시
- 완료 시 Etherscan 트랜잭션 링크 제공
- 실패 시 에러 메시지 표시

### 2.4 비기능 요구사항

| 구분 | 요구사항 |
|------|----------|
| 성능 | 읽기(view 함수) 응답: 1초 이내, 쓰기 트랜잭션: MetaMask 서명 후 Sepolia 블록 확인까지 ~15초 허용 |
| 보안 | 사용자는 본인 지갑 주소(msg.sender)의 데이터만 조회/수정/삭제 가능 |
| 가용성 | Sepolia RPC(Infura/Alchemy) 가용성에 의존 |
| 호환성 | Chrome + MetaMask 확장 환경 지원 (모바일은 v1 범위 외) |
| 확장성 | 컨트랙트 구조를 Polygon 등 다른 EVM 체인으로 이식 가능하게 설계 |

### 2.5 제약사항 및 가정

| 구분 | 내용 |
|------|------|
| 비용 | Sepolia 테스트넷 사용 → 실제 ETH 비용 없음, Sepolia ETH는 Faucet으로 조달 |
| 가스 제한 | 온체인 저장 특성상 문자열 길이 제한 필수 (제목 100자, 설명 500자 이내) |
| 프라이버시 | 온체인 데이터는 공개됨 — 민감정보 입력 금지 안내 표시 |
| 지갑 | MetaMask 설치된 환경에서만 동작 |
| 다중 계정 | 각 지갑 주소별로 독립적인 데이터 공간 |

### 3.2 스마트 컨트랙트 명세

#### 3.2.1 컨트랙트 개요

| 항목 | 내용 |
|------|------|
| 컨트랙트명 | `TodoList` |
| 솔리디티 버전 | `^0.8.20` |
| 라이선스 | MIT |
| 배포 네트워크 | Ethereum Sepolia Testnet |

#### 3.2.2 데이터 구조

```
enum Priority {
    LOW,       // 0
    MEDIUM,    // 1
    HIGH       // 2
}

struct Category {
    uint256 id;
    string  name;
    bool    isActive;
}

struct Todo {
    uint256  id;
    string   title;       // 최대 100자
    string   description; // 최대 500자
    uint256  categoryId;
    Priority priority;
    uint256  deadline;    // Unix timestamp (0 = 마감일 없음)
    bool     isCompleted;
    bool     isActive;    // 논리적 삭제용 플래그
    uint256  createdAt;   // block.timestamp
    uint256  updatedAt;   // block.timestamp
}
```

#### 3.2.3 상태 변수

| 변수명 | 타입 | 설명 |
|--------|------|------|
| `_todoCounter` | `uint256` | 전역 Todo ID 카운터 |
| `_categoryCounter` | `uint256` | 전역 Category ID 카운터 |
| `_userTodos` | `mapping(address => Todo[])` | 주소별 Todo 배열 |
| `_userCategories` | `mapping(address => Category[])` | 주소별 Category 배열 |
| `_todoIndexById` | `mapping(address => mapping(uint256 => uint256))` | Todo ID → 배열 인덱스 |
| `_categoryIndexById` | `mapping(address => mapping(uint256 => uint256))` | Category ID → 배열 인덱스 |

#### 3.2.4 함수 명세

**카테고리 관련**

| 함수명 | 접근 제어 | 반환값 | 설명 |
|--------|-----------|--------|------|
| `addCategory(string name)` | external | `uint256 categoryId` | 카테고리 추가 |
| `deleteCategory(uint256 categoryId)` | external | — | 카테고리 논리 삭제 |
| `getCategories()` | external view | `Category[]` | 내 카테고리 전체 조회 |

**Todo 관련**

| 함수명 | 접근 제어 | 반환값 | 설명 |
|--------|-----------|--------|------|
| `addTodo(string title, string description, uint256 categoryId, Priority priority, uint256 deadline)` | external | `uint256 todoId` | Todo 추가 |
| `updateTodo(uint256 todoId, string title, string description, uint256 categoryId, Priority priority, uint256 deadline)` | external | — | Todo 수정 |
| `deleteTodo(uint256 todoId)` | external | — | Todo 논리 삭제 |
| `toggleComplete(uint256 todoId)` | external | — | 완료/미완료 토글 |
| `getTodos()` | external view | `Todo[]` | 내 Todo 전체 조회 (isActive=true만) |
| `getTodoById(uint256 todoId)` | external view | `Todo` | 특정 Todo 단건 조회 |

#### 3.2.5 이벤트(Events)

| 이벤트명 | 파라미터 | 발생 시점 |
|----------|----------|-----------|
| `TodoAdded` | `address indexed owner, uint256 indexed todoId` | Todo 생성 시 |
| `TodoUpdated` | `address indexed owner, uint256 indexed todoId` | Todo 수정 시 |
| `TodoDeleted` | `address indexed owner, uint256 indexed todoId` | Todo 삭제 시 |
| `TodoCompleted` | `address indexed owner, uint256 indexed todoId, bool isCompleted` | 완료 토글 시 |
| `CategoryAdded` | `address indexed owner, uint256 indexed categoryId` | 카테고리 생성 시 |
| `CategoryDeleted` | `address indexed owner, uint256 indexed categoryId` | 카테고리 삭제 시 |

#### 3.2.6 입력 유효성 검증 (require 조건)

| 조건 | 에러 메시지 |
|------|-------------|
| `title` 길이 > 0 | `"Title cannot be empty"` |
| `title` 길이 ≤ 100 | `"Title too long"` |
| `description` 길이 ≤ 500 | `"Description too long"` |
| `categoryId`가 존재하는 카테고리 | `"Invalid category"` |
| `deadline`이 0이거나 미래 시점 | `"Invalid deadline"` |
| `todoId`가 존재하는 Todo | `"Todo not found"` |
| `isActive == true` (삭제된 항목 접근 방지) | `"Todo is deleted"` |
| `msg.sender == todo.owner` (암묵적, mapping 구조로 보장) | — |

#### 3.2.7 가스 최적화 고려사항

- 문자열은 `string` 타입 사용 (짧은 문자열의 경우 `bytes32` 고려했으나, 가독성 우선으로 `string` 채택)
- 배열 순회 대신 `mapping`으로 인덱스 관리하여 O(1) 접근
- `view` 함수는 가스비 없음 — 조회는 최대한 뷰 함수로 처리
- 삭제 시 `delete` 대신 `isActive = false` 플래그로 처리 (배열 재정렬 가스비 회피)

#### 3.3.5 트랜잭션 처리 플로우

```

    │

    │
    ▼
MetaMask 팝업 → 서명 요청

                   │
                   ├─ 성공 → UI 업데이트 + Etherscan 링크 토스트
                   └─ 실패 → 에러 토스트 표시
```

### 3.4 보안 요구사항

| 항목 | 요구사항 |
|------|----------|
| 접근 제어 | 모든 쓰기 함수는 `msg.sender` 기준으로 본인 데이터만 조작 가능 |
| Reentrancy | 외부 호출 없으므로 reentrancy 위험 없음 (Checks-Effects-Interactions 패턴 준수) |
| Integer Overflow | Solidity 0.8.x 기본 overflow 방지 내장 |
| 입력 검증 | 문자열 길이 및 유효한 enum 값 검증 |
| 프라이버시 경고 | UI에서 "온체인 데이터는 공개됩니다" 안내 문구 상시 표시 |
| XSS | React의 기본 이스케이프 처리로 방어 |

## 5. 구현 계획

> **기준**: 1인 개발자, 각 Phase ≈ 1 man/day (약 6~8시간)
> **각 Phase 포함 사항**: 구현 → 코드 자체 리뷰 → 단위 테스트 → 사이드 이펙트 검증 → 이전 Phase 전체 회귀 테스트

---

### Phase 1 — 개발 환경 구성 + 프로젝트 스켈레톤 (Day 1)

**목표**: 빌드 가능한 초기 프로젝트 구조 완성

**구현**
- [ ] Hardhat 프로젝트 초기화 (`hardhat init`)
- [ ] `TodoList.sol` 파일 생성 + 데이터 구조 선언 (struct, enum, mapping, event)
- [ ] Vite + React 프로젝트 초기화
- [ ] ethers.js, Tailwind CSS 의존성 설치
- [ ] `.env.example` 파일 작성 (환경 변수 목록 정의)
- [ ] Git 리포지토리 초기화 및 `.gitignore` 설정

**코드 리뷰 체크리스트**
- 데이터 구조가 SRS 3.2.2와 일치하는가
- 민감 정보(개인키, RPC URL)가 `.gitignore`에 포함되었는가
- 솔리디티 버전 pragma 설정이 올바른가

**테스트**
- `npx hardhat compile` → 오류 없이 컴파일 성공 확인
- React 앱 `npm run dev` → 로컬 서버 정상 구동 확인

**사이드 이펙트 검증**
- 해당 없음 (초기 단계)

---

### Phase 2 — 스마트 컨트랙트 핵심 로직 구현 (Day 2)

**목표**: CRUD + 카테고리 + 우선순위 전체 함수 구현

**구현**
- [ ] `addCategory`, `deleteCategory`, `getCategories` 구현
- [ ] `addTodo` 구현 (입력 유효성 검증 포함)
- [ ] `updateTodo` 구현
- [ ] `deleteTodo` (논리 삭제) 구현
- [ ] `toggleComplete` 구현
- [ ] `getTodos`, `getTodoById` 구현
- [ ] 모든 이벤트 emit 추가
- [ ] require 조건 SRS 3.2.6 기준으로 전부 구현

**코드 리뷰 체크리스트**
- msg.sender 기반 접근 제어가 모든 쓰기 함수에 적용되었는가
- 문자열 길이 제한 require가 누락된 곳은 없는가
- 이벤트가 모든 상태 변경 후 emit되는가
